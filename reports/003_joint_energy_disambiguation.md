# MVP 0.3 Joint Energy Disambiguation

Generated by:

```bash
PYTHONPATH=src:. python3 experiments/joint_energy_disambiguation.py
```

## Purpose

MVP 0.2 showed a concrete failure mode: if content-similar anchors are too close
in FHRR space, post-content temporal recall fails because content settling blends
the wrong anchors before the temporal channel is read.

This experiment tests the first fix: score candidate anchors by both content
compatibility and temporal-context compatibility.

## Setup

- `dim = 512`
- `window = 2`
- `family_noise = 0.10`
- `content_beta = 100.0`
- `temporal_beta = 100.0`
- Query anchor: `stair`
- Temporal cue: bundle of `slip` and `doctor`

The mobility family `stair`, `ladder`, `ramp`, `step`, and `escalator` is almost
identical in content space. Only `stair` has the lived temporal neighborhood
containing `slip` and `doctor`.

## Result

| Method | Temporal-neighbor Recall@4 |
|---|---:|
| Content nearest-neighbor | 0.000 |
| Post-content temporal recall | 0.000 |
| Joint content+temporal recall | 1.000 |

The content baseline retrieved:

```text
ladder, escalator, step, ramp
```

The post-content temporal read retrieved unrelated blended contexts:

```text
letter, music, cart, dance
```

The joint read retrieved the lived neighbors:

```text
slip, table, candle, doctor
```

The joint top anchor was `stair`.

## Interpretation

This rescues the tight-family failure from MVP 0.2. The lesson is architectural:
temporal association should not be read only after content retrieval has already
settled. It needs to be part of the settling energy itself.

This is still a hand-specified joint score, not the final no-controller form.
But it points in the right direction: content and temporal compatibility can be
terms in one energy field, so the correct anchor is the state that jointly
satisfies both.
